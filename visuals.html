Visuals

Images

Images are totally great as long as they are backed up with the appropriate underlying code/text and are not the sole mode of accessing information.


alt="George Washington">

For an image like this which is usually primarily decorative, the alternative text can just say "George Washington".


But for an image such as a map or graph that contains information, a more detailed description is needed.



alt="A picture of Spiderman laying in bed captioned I like to play this game called nap roulette I take a nap and not set an alarm. Will it be 20 minutes or 4 hours? Nobody knows. It's risky and I like it">

Do not neglect to describe images that are just for fun. Screen reader users want to share in the joke. Besides, screen readers are boring and, as you have seen, not always easy to work with, so it is nice to access something fun.



Interactive stuff

Some tools are great because they give automatic feedback immediately. However, they are prone to accessibility issues. Most of these issues could be resolved by making sure the graphics are backed by text. In the case of feedback on if an answer is correct or incorrect, delivering the feedback auditorily as well as visually, or at least in text as well, would allow screen readers to access that feedback. If feedback is given in multiple formats, it is both more accessible and more engaging.

Emojis

Emojis are a great way to express emotion and convey meaning quickly. One or two are great even with a screen reader. It can usually describe them automatically, So they still add some fun variety. Putting a ton of them together though is annoying! Screen readers describe in words what the emoji looks like. While all emojis visually take up about the same amount of space, some of the descriptions the screen reader gives can be quite long.



Links

When reading visually, links are easy to skip over, wherever they are. A screen reader can not skim read. If the link is in the middle of a text message, a screen reader user has to listen to the entire link to get to the last part of the message. So separate the links or put them at the end. It is really annoying to hear a robot read out a long link to get to the stuff after it. It makes you wonder: do I really want to know what the end of this is or can I just skip it? 1 second of robot reading later: Skip

Change of appearance is not a good way to differentiate information on its own. Certain HTML tags can help with this. Screen readers cannot help with things that rely solely on CSS differences and not on HTML text or tags. For example, to indicate that text is bold in a way that screen readers can pick up on is to use the strong HTML tag.



Key Command Consistancy

Outlook and Gmail, Google Docs and Word, Chrome and Safari are all examples of the same type of thing but which have very different Keyboard commands to do those things. Since the screen readers rely on keyboard navigation instead of mouse, this means that a screen reader user has to memorize all of the keyboard commands for the thing they're using. Trying to do that for each different app and website is not effective or feasible. So when making a website, work with the native interfaces of the user's computer rather than making your own that will just make things more confusing.
